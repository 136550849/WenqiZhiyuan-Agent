### 一、OpenClaw 系统架构深度理解

#### 1.1 核心定位

> **OpenClaw** = 自托管网关 + 多通道消息 + AI 智能体

**关键特性**:
- ✅ 自托管：运行在自己的硬件上，自己掌控
- ✅ 多通道：WhatsApp, Telegram, Discord, iMessage 单一网关
- ✅ 智能体原生：为编码智能体设计，支持工具使用、会话、记忆
- ✅ 开源：MIT 许可，社区驱动

#### 1.2 架构流程

```
聊天应用 (WhatsApp/Telegram/Discord)
        ↓
    Gateway (网关)
        ↓
   ┌────┼────┐
   ↓    ↓    ↓
  Pi   CLI  Web Control UI
Agent      macOS App
           iOS/Android Nodes
```

**Gateway 是单一事实源**：负责会话、路由、通道连接

#### 1.3 Pi 智能体集成架构

**集成方式**: 嵌入式 (非 RPC/子进程)

```typescript
createAgentSession({
  cwd: workspace,
  agentDir,
  authStorage,
  modelRegistry,
  model: "qwen/qwen3.5-plus",
  tools: [...],
  customTools: [...],
})
```

**核心优势**:
- 完全控制会话生命周期
- 自定义工具注入
- 系统提示词定制
- 会话持久化 + 分支/压缩
- 多账户认证配置轮换
- 提供商无关的模型切换

#### 1.4 工具架构

**工具流水线**:
1. **基础工具**: Pi 的 codingTools (read, bash, edit, write)
2. **自定义替换**: OpenClaw 替换 bash 为 exec/process

4. **通道工具**: Discord/Telegram/Slack/WhatsApp 特定动作

### 二、日志系统深度理解

#### 2.1 日志位置

**默认位置**: `/tmp/openclaw/openclaw-YYYY-MM-DD.log`

**配置覆盖**:
```json
{
  "logging": {
    "file": "/path/to/openclaw.log"
  }
}
```

#### 2.2 日志读取方式

**CLI 实时追踪** (推荐):
```bash
openclaw logs --follow
```

**输出模式**:
- TTY 会话：彩色结构化日志
- 非 TTY 会话：纯文本
- `--json`: JSONL 格式
- `--plain`: 强制纯文本
- `--no-color`: 禁用 ANSI 颜色

**Control UI**: Web 界面 Logs 标签页

#### 2.3 日志配置

```json
{
  "logging": {
    "level": "info",
    "file": "/tmp/openclaw/openclaw-YYYY-MM-DD.log",
    "consoleLevel": "info",
    "consoleStyle": "pretty",
    "redactSensitive": "tools",
    "redactPatterns": ["sk-.*"]
  }
}
```

**日志级别**:
- `logging.level`: 文件日志级别
- `logging.consoleLevel`: 控制台级别
- 环境变量 `OPENCLAW_LOG_LEVEL` 优先
- CLI 选项 `--log-level <level>` 覆盖环境变量

**控制台样式**:
- `pretty`: 人类友好，彩色，带时间戳
- `compact`: 紧凑输出 (适合长会话)
- `json`: 每行 JSON (适合日志处理器)

**脱敏**:
- `logging.redactSensitive`: `off` | `tools` (默认)
- `logging.redactPatterns`: 正则表达式列表

#### 2.4 诊断 + OpenTelemetry

**诊断事件目录**:

**模型使用**:
- `model.usage`: tokens, cost, duration, context, provider/model/channel

**消息流**:
- `webhook.received`: webhook 入口
- `webhook.processed`: webhook 处理 + 耗时
- `webhook.error`: webhook 错误
- `message.queued`: 消息入队
- `message.processed`: 结果 + 耗时

**队列 + 会话**:
- `queue.lane.enqueue`: 队列入队 + 深度
- `queue.lane.dequeue`: 队列出队 + 等待时间
- `session.state`: 会话状态转换
- `session.stuck`: 会话卡住警告
- `run.attempt`: 重试/尝试元数据

**OpenTelemetry 导出**:

### 三、TTS (文本转语音) 系统

#### 3.1 支持的服务

| 服务 | API Key | 默认 | 说明 |
|------|---------|------|------|
| **ElevenLabs** | 需要 | 可选 | 主要或备用 |
| **OpenAI** | 需要 | 可选 | 主要或备用 |
| **Edge TTS** | 不需要 | ✅ 默认 | 微软 Edge 在线服务 |

#### 3.2 TTS 配置

**最小配置**:
```json
{
  "messages": {
    "tts": {
      "auto": "always",
      "provider": "elevenlabs"
    }
  }
}
```

**Edge TTS 配置** (无 API Key):
```json
{
  "messages": {
    "tts": {
      "auto": "always",
      "provider": "edge",
      "edge": {
        "enabled": true,
        "voice": "en-US-MichelleNeural",
        "lang": "en-US",
        "outputFormat": "audio-24khz-48kbitrate-mono-mp3",
        "rate": "+10%",
        "pitch": "-5%"
      }
    }
  }
}
```

**自动 TTS 模式**:
- `off`: 禁用
- `always`: 总是启用
- `inbound`: 仅在收到语音消息后
- `tagged`: 仅当回复包含 `[[tts]]` 标签

#### 3.3 输出格式

| 通道 | 格式 | 说明 |
|------|------|------|
| **Telegram** | Opus voice-note | 48kHz/64kbps，圆形气泡 |
| **其他** | MP3 | 44.1kHz/128kbps |
| **Edge TTS** | 可配置 | 默认 24kHz/48kbps MP3 |

#### 3.4 自动 TTS 行为

```
回复 → TTS 启用？
  否 → 发送文本
  是 → 有媒体/MEDIA:/短回复？
        是 → 发送文本
        否 → 长度 > 限制？
              否 → TTS → 附加音频
              是 → 摘要启用？
                    否 → 发送文本
                    是 → 摘要 → TTS → 附加音频
```

#### 3.5 斜杠命令

```bash
/tts off          # 禁用
/tts always       # 总是启用
/tts inbound      # 仅收到语音后
/tts tagged       # 仅带标签时
/tts status       # 查看状态
/tts provider openai  # 设置提供商
/tts limit 2000   # 设置长度限制
/tts summary off  # 禁用摘要
/tts audio Hello  # 一次性生成音频
```
